# DECISIONS

This document records key design decisions for the DrawTabData project.


## 2026-04-16: Per-unit defects on inventory records, controlled by a vocabulary file

### Context
Some physical units in inventory are defective — sometimes completely
dead, sometimes broken in specific ways (e.g. pen reports a max pressure
value far below the expected range). The dataset had no way to record
this, and "defective" is a property of the specific unit, not of the
model, so it doesn't belong on tablet or pen model records.

### Options considered
1. A `Condition` wrapper object with a `Status` enum (`ok`, `degraded`,
   `defective`, `dead`) and a `Defects` array inside it.
2. A simple `IsDefective` boolean plus free-text `DefectNotes`.
3. Just a `Defects` array on inventory records, with no wrapper or
   status field. Each entry is `{ Kind, Notes }`. The `Kind` values are
   defined in a separate vocabulary file.

### Decision
Option 3.

- `Defects` lives directly on each pen or tablet inventory record. It
  is optional — absent or empty means the unit has no recorded defects.
- Each entry is `{ "Kind": "...", "Notes": "..." }`. Multiple entries
  are allowed when a unit fails in multiple ways.
- The full list of allowed `Kind` values lives in
  `data/reference/defect-kinds.json`, keyed on `Kind` with an
  `AppliesTo` scope (`"pen"`, `"tablet"`, or both) and a human-readable
  `Description`.

### Rationale
- A `Status`/`Condition` wrapper is redundant: presence of entries in
  `Defects` is itself the signal that something is wrong. One less
  field to keep in sync.
- A controlled vocabulary in a separate file lets every consumer
  (this inventory app, DrawTabDataExplorer, any future tool) agree on
  the same Kind values and offer consistent filtering and grouping.
- The `{ Kind, Notes }` tuple is structured enough to be queryable
  ("show all pens with pressure issues") without locking us into
  fields we may not always have (measured values, severity enums).
- Severity is captured descriptively in `Notes`, not as a separate
  enum, until a concrete consumer needs to filter by severity.

### Fields affected
- `InventoryPens[].Defects` — optional array of `{ Kind, Notes }`
- `InventoryTablets[].Defects` — optional array of `{ Kind, Notes }`
- new: `data/reference/defect-kinds.json` — vocabulary file with
  `{ DefectKinds: [{ Kind, AppliesTo, Description }] }`

### Consumer impact
- DrawTabInventory renders a Defects column and offers a
  "With defects / Without defects / All" filter.
- Other consumers (DrawTabDataExplorer, PenPressureData, Wacom-Driver-List)
  are unaffected until they opt in — the field is additive.

See docs/DEFECTS.md for the full description and the current Kind list.


## 2026-04-13: Model.IncludedPen stores pen EntityIds, not PenIds

### Context
`Model.IncludedPen` originally stored short `PenId` strings (e.g. `"X3ELITE"`).
Consumers that needed to link to a pen detail page or look up pen metadata had
to load all pen records and build a `PenId → EntityId` map first.

### Decision
Store the full pen `EntityId` (e.g. `"XPPEN.PEN.X3ELITE"`) in `Model.IncludedPen`
instead of the bare `PenId`. This lets consumers build links and look up pen data
directly without a separate resolution step.

### Migration
A one-time script converted all 292 affected tablet records across 5 brand files.
5 PenId mismatches (non-standard casing or missing hyphens) were corrected
manually. 1 record (`9494G` / `PD04B`) was left unchanged as that pen is not
yet in the dataset.

### Fields affected
- `Model.IncludedPen` — array of pen EntityId strings instead of PenId strings

### Impact on consumers
- `buildPenNameMap` now keys by `EntityId` instead of `PenId`
- Pen detail page: `IncludedPen` filter compares to `found.EntityId`
- Tablet list view: `cellLinks` for `ModelIncludedPen` uses the stored
  value directly as the URL path; no intermediate `penIdToEntityId` map needed


## 2026-04-01: Complex fields for multi-part values

### Context
Some fields stored multi-part values as delimited strings (e.g. "287.00 x 186.00 x 9.00").
This made partial data awkward (trailing " x " when a component is missing), required
string parsing for programmatic use, and made validation harder.

### Decision
Multi-part fields use nested JSON objects with named numeric properties instead of
delimited strings. Consumers access the data through a TypeScript library that returns
typed objects, so the added JSON complexity is not exposed directly.

### Fields affected
- PhysicalDimensions  -> { "Width": number, "Height": number, "Depth": number }
- DigitizerDimensions -> { "Width": number, "Height": number }
- DisplayResolution   -> { "Width": number, "Height": number }

### Key details
- Property names use long form: Width, Height, Depth (not W, H, D)
- Values are numbers, not strings
- Missing components are omitted from the object (e.g. if Depth is unknown,
  the key is simply absent)


## 2026-04-01: DisplayColorGamuts as a complex field with normalized gamut names

### Context
DisplayColorGamuts was a comma-delimited string of gamut:coverage pairs with
inconsistent naming (e.g. "sRGB" vs "SRGB", "DCI-P3" vs "DCIP3" vs "DCIPR"),
inconsistent delimiters, and some corrupted entries using Chinese commas.

### Decision
Convert DisplayColorGamuts to a nested object keyed by canonical gamut name,
with numeric percentage values.

### Canonical gamut names (all uppercase)
- SRGB       (normalizes: sRGB, SRGB)
- ADOBERGB   (normalizes: AdobeRGB, ADOBERGB, Adobe)
- DCIP3      (normalizes: DCI-P3, DCIP3, DCIPR)
- DISPLAYP3  (normalizes: DisplayP3, DISPLAYP3)
- NTSC
- REC709

Note: DCIP3 and DISPLAYP3 are distinct color spaces.

### Format
{ "SRGB": 120, "AdobeRGB": 90, "NTSC": 88 }
Values are numbers representing coverage percentage (without % sign).

### Data corrections applied
Two HUION entries had Chinese commas and swapped gamut/percentage:
- "NTSC，127%:90%" interpreted as NTSC:90%, SRGB:127%
- "NTSC，130%:92%" interpreted as NTSC:92%, SRGB:130%
This matches the pattern of other HUION entries in the same product line.


## 2026-04-13: Nested JSON groups for tablet records

### Context
Tablet records had grown to ~40 flat top-level fields with name prefixes
(e.g. `DigitizerDimensions`, `DisplayPixelDimensions`, `ModelLaunchYear`).
This made new-field placement ambiguous, made the JSON verbose, and made
it harder to see at a glance which fields belonged to which system.

### Decision
Reorganise tablet records into six nested groups, stripping the prefix
within each group:

- **Meta** — identity and timestamps (`EntityId`, `_id`, `_CreateDate`, `_ModifiedDate`)
- **Model** — product classification (`Brand`, `Id`, `Name`, `Type`, `LaunchYear`, ...)
- **Digitizer** — drawing surface (`Dimensions`, `PressureLevels`, `Tilt`, ...)
- **Display** — screen, present on PENDISPLAY and STANDALONE only (`PixelDimensions`, `Brightness`, ...)
- **Physical** — physical form (`Dimensions`, `Weight`, ...)
- **Standalone** — compute/battery/connectivity, present on STANDALONE only (`OS`, `RAM`, `Storage`, ...)

The PENTABLET-cannot-have-Display rule and the STANDALONE-only rule for the
Standalone group are enforced in the Valibot schema via `v.rawCheck`.

### Migration
A one-time migration script (`scripts/migrate-flat-to-nested.ts`) transformed
all 306 tablet records across 5 brand files. The `npm run data-quality`
validator confirmed 0 issues after migration.

### Consumer impact
Only DrawTabDataExplorer required code updates. PenPressureData, Wacom-Driver-List,
and DrawTabInventory do not consume the Tablet type.


## 2026-04-02: Pre-filtered navigation via URL query parameters

### Context
The data quality view identifies sets of entities matching certain criteria
(e.g. "Wacom tablets with no pen compatibility data"). Users want to click
through to see those entities in the entity explorer with filters pre-applied.

### Options considered
1. Full URL-encoded pipeline state (filter + sort + columns in URL)
2. Named saved views created on the fly
3. Inline expansion within data quality page
4. Simple pre-filtered links via query parameters

### Decision
Option 4: simple query parameter filtering. Entity pages read `?filter=`
params from the URL on load and apply them as initial filters.

Format: `?filter=Field:operator:value` (e.g. `?filter=Brand:==:WACOM`)

### Rationale
- Covers the primary use case (filtered lists from data quality)
- Works on GitHub Pages (query params are client-side)
- Simple to implement — no full URL state encoding needed
- Shareable links
- Can upgrade to full URL state later if needed
